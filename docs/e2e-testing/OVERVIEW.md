# AI駆動E2Eドキュメント設計の全体像

本ドキュメントは、Chrome DevTools MCP と日本語Gherkinを用いたAI自律E2E実行のために構築したディレクトリおよび役割分担を俯瞰的に説明する。RUNBOOK を軸に、ガイドライン・機能Feature・長尺シナリオ・フォールバックスニペットを組み合わせて AI が迷わず実行できるように設計されている。

## コア構造
```
docs/e2e-testing/
├─ RUNBOOK.md
├─ OVERVIEW.md (本ドキュメント)
├─ guidelines/
│  └─ ai-rules.md
├─ features/
├─ scenarios/
└─ snippets/
```

### RUNBOOK.md
- AIが最初に読み込む実行基点。
- 目的、前提条件、実行フロー、フォールバック規約、成功/失敗基準、代表タグを列挙し、どのように行動すべきかを明示する。
- 実行時はRUNBOOKに従い、必要な詳細は他ファイルへディープリンクする構造を採用。

### guidelines/ai-rules.md
- 環境変数 (`env`) と運用ポリシー (`policy`)、自然言語→`feature@tag` 対応表 (`mapping`) を集約。
- MCP固有の設定値のうち、実行判断に本質的でないものは排除してシンプルに保ち、AIが参照すべき要素に集中させる。
- RUNBOOK の「ガイドライン読込」ステップで必ず適用される想定。

### features/
- 単機能の操作・検証を日本語Gherkinで記述するディレクトリ。
- `params` セクションで共通のURLやロケールを宣言し、Backgroundで初期状態を整備。
- タグ（例: `@ログイン`, `@カート追加`）を付与して、RUNBOOKやマッピング経由で呼び出しやすくする。

### scenarios/
- 複数のFeatureを連携させたユーザージャーニーを記述。
- 各ステップで `feature "../features/<name>.feature"` のシナリオを相対パスで呼び出し、ディレクトリ内の関係性を明示する。
- 長尺のE2E検証に必要なログ取得や検証をまとめて管理。

### snippets/
- 通常操作がリトライ上限まで失敗したときにのみ使用するフォールバックコードを配置。
- RUNBOOKおよびguidelinesでフォールバックポリシーを明文化し、無秩序な利用を防ぐ。

## 設計意図
1. **単一の起点**: RUNBOOKが全体の流れを支配し、AIは迷わず手順を再現できる。
2. **責務分離**: ガイドライン、機能Feature、長尺シナリオ、フォールバックコードを分離することで、変更の影響範囲が明確になる。
3. **再利用性**: Featureをタグで識別し、Scenarioから相対パスで参照することで、複数のジャーニーにまたがる再利用を容易にする。
4. **失敗時の統制**: スニペットは最後の手段として明文化されており、成功時は通常操作のみで完走する前提を保つ。

## 運用フローの連動
1. AIは `RUNBOOK.md` を読み、前提・手順・評価基準を理解する。
2. 次に `guidelines/ai-rules.md` を参照し、対象環境やマッピング情報を適用する。
3. 利用者の意図に合わせて `features/` あるいは `scenarios/` を選択。Scenario内では相対パスによってFeatureを連結する。
4. 実行中に操作が失敗した場合のみ `snippets/` を参照し、RUNBOOK記載のポリシーに沿ってフォールバックする。

この構造により、ドキュメントのどこから読んでも最終的にRUNBOOKに立ち返る導線が張られ、AIが自律的にE2Eテストを遂行できるようになっている。

## 初期設計との整合性チェック
- **RUNBOOK起点の実行**: `RUNBOOK.md` が目的・前提・実行フロー・成功/失敗基準・フォールバック方針・代表タグを網羅し、初期設計で要求された「唯一の実行起点」を満たしている。
- **ガイドライン集約**: `guidelines/ai-rules.md` に `env` / `mcp` / `policy` / `mapping` を統合し、旧plan系資料を排しつつ初期設計どおりの参照順序を維持している。
- **日本語Gherkinによる機能定義**: `features/` 配下の各Featureが `params` と Background を備え、待機・ロケータ優先度・API検証・スニペット呼び出しを日本語で明示しており、フォールバックは明示された場合に限定される。
- **シナリオの相対参照**: `scenarios/` 内の長尺シナリオが `feature "../features/..."` で相対参照し、複数機能の連結という初期設計の要件を満たす。
- **フォールバック統制**: `snippets/` は通常操作が失敗した際の一度限りの救済として位置づけられ、RUNBOOKおよびガイドライン双方で使用条件が揃っている。

上記のチェックを通じ、初回の設計要求で定義された構造・責務・運用ポリシーが現在のディレクトリとドキュメントに反映されていることを確認した。
