# Chrome DevTools MCP E2E テストフレームワーク

このドキュメントは、Chrome DevTools MCP サーバーを使用した E2E テストの実行手順を定義しています。

## 概要

各テストケースは独立したGherkinファイルで定義され、このドキュメントが実行フローを管理します。**テスト実行がこのドキュメントの主目的**であり、環境設定とURL情報は参考資料として位置付けられています。

## 基本的なテスト実行フロー

### 1. 前提条件
- Chrome DevTools MCP サーバーが稼働していることを確認
- 対象環境のアプリケーションがアクセス可能であることを確認
- **テストアカウント情報の取得**：
  ```
  テストアカウントは事前に準備するか、セキュアストレージから取得する必要があります。

  取得すべき情報：
  - test-user@example.com のパスワード
  - admin-user@example.com のパスワード
  - その他のテストアカウント認証情報

  一度取得した情報はセッション中再利用する
  ```

### 2. 実行手順
1. **ブラウザ起動**: 複数タブで対象環境アプリを開く
2. **認証確認**: 各アプリの認証状態を即座に確認（未認証なら終了）
3. **テスト実行**: 指定されたテストケースを順次実行
4. **結果検証**: 期待結果との比較
5. **レポート生成**: 実行結果を記録

### 3. 重要な実行ルール
- **認証失敗時の即座終了**: 認証が完了していない場合、テストを即座に終了しユーザー認証を要求
- **独立実行**: 各テストは独立して実行可能でなければならない
- **データ整合性**: テスト実行前後のデータ整合性を維持

## 参考情報

### テスト環境URL

**使用方法**: テスト実行指示に応じて、以下のテーブルから適切な環境とアプリケーションタイプのURLを選択してください。

| 環境 | アプリケーションタイプ | URL | 説明 |
|------|------------------------|-----|------|
| ローカル | メインアプリ | http://localhost:3000 | ローカル開発メインアプリケーション |
| 開発 | メインアプリ | https://dev.example.com | 開発環境 |
| ステージング | メインアプリ | https://staging.example.com | ステージング環境 |

### ブラウザ操作基本ルール

- **タブ管理**: 異なるアプリケーションビュー（ユーザー、管理者等）を一つのブラウザウィンドウの異なるタブで管理
- **認証確認**: 各アプリへのアクセス時に認証状態を即座に確認（必須）
- **認証失敗**: 未認証の場合、テストを即座に終了しユーザー認証を要求
- **タブ切り替え**: 操作対象に応じて適切なタブに切り替え

### ページ読み込み確認戦略

アプリケーションの応答が遅い場合、以下の順序でページ読み込み完了を確認：

#### 1. ネットワーク状態確認
```
list_network_requests → 進行中のAPIリクエストがないことを確認
```

#### 2. React コンポーネント読み込み状態確認
```
evaluate_script → ローディングインジケーターの確認
take_snapshot → ローディング要素（スピナー、プログレスバー等）の消失確認
```

#### 3. DOM要素安定性確認
```
take_snapshot → 対象要素が存在し操作可能であることを確認
```

#### 4. 待機戦略優先順位
1. **推奨**: ネットワーク + React状態確認
2. **代替**: `wait_for` 特定要素（要素が確実に存在する場合のみ）
3. **非推奨**: 固定時間待機

## テスト実行フロー

1. **テストアカウント情報取得**
   - セッション開始時に一度実行
   - セッション中は取得した情報を再利用
2. ブラウザ起動と必要なタブを開く
3. 認証状態確認（未認証なら即座に終了）
4. 指定された操作を順次実行
5. 結果検証とレポート

## エラーハンドリング

### 認証エラー
- 認証が確認できない場合、テストを即座に終了
- エラーメッセージを出力し、ユーザー認証を要求

### 要素が見つからない
- 指定された要素が見つからない場合、詳細なエラー情報を出力
- 現在の状態を確認するためのスクリーンショットを取得

### ページ読み込み遅延
- **ネットワークタイムアウト**: 長時間応答しないAPIリクエストを特定し詳細を記録
- **コンポーネント未レンダリング**: Reactコンポーネントが未レンダリングの場合、console.errorsを確認
- **JavaScriptエラー**: `list_console_messages`でブラウザエラーを収集・分析

### タイムアウト
- 各操作に適切なタイムアウトを設定
- タイムアウト時はスクリーンショットで現在の状態を記録
- ネットワーク状態とコンソールエラーを同時に記録

## 重要な注意事項

- 各テストは独立して実行可能でなければならない
- テスト実行前後のデータ整合性を維持
- 本番環境での実行は避ける
- テストデータの適切なクリーンアップを実施
